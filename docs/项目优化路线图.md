# Graph RAG 项目优化路线图：从教程二次开发 → 大厂级生产项目

> 目标：把这个项目从"功能跑通的二次开发"升级成"能在简历上深挖、能扛面试官追问、有量化指标的生产级 AI 服务"。
> 原则：**每个阶段 = 一个简历亮点 + 一组面试深挖预案**。不做为了堆技术而堆的事。

## 二、各阶段详解（文件级、可执行）

### 🔧 P0 工程化重构（地基，先做）
**目的**：把"脚本项目"变成"工程项目"，后面所有功能都长在这个骨架上。

具体改动：
- **统一配置**：新建 `code/config/settings.py`（用 `pydantic-settings`），把散在 `.env` / `config.json` / `config.py` / 硬编码的 LLM 配置收口。**修掉当前的混乱**：`generation_integration.py` 用 `MOONSHOT_API_KEY` 却指向 `bigmodel.cn`，模型名在 kimi/glm-4-flash 之间不一致。
- **FastAPI 服务层**：新建 `code/api/server.py`，把 `AdvancedGraphRAGSystem` 包成 HTTP 服务，先暴露 `POST /ask`（非流式）和 `GET /health`。CLI（`main.py`）保留，但变成可选。
- **日志规范**：把满屏 `print()` 换成结构化 `logging`（JSON 格式），每个请求带 `request_id`。
- **目录重构**：`code/` 下分 `core/`（系统）、`rag_modules/`（现有）、`api/`、`cache/`（P1）、`eval/`（P2）、`infra/`（P6）。

**简历怎么写**：用 pydantic-settings 统一多源配置；FastAPI 暴露 RESTful 接口；结构化日志 + request_id 全链路追踪。
**深挖预案**：配置为什么要统一（多环境/密钥管理）？FastAPI 为什么不用 Flask（async/异步 SSE 原生支持）？

### 📊 P2 RAG 评测体系（最关键）
**目的**：让简历从"我做了 X"升级到"我证明 X 让指标从 A 到 B"。这是 demo 与生产级的分水岭，也是 RAG 面试必问。

> **方法论演进（重要，本身就是面试素材）**：初版 P2 只做"系统内 6 路策略横向消融"，实测发现 graph_rag 在"菜谱名召回"指标上恒为 0——不是它没用，而是**召回指标对返回子图/路径的 graph_rag 不公平（度量错位）**；且测试集 63% 是简单查菜查询、既缺 graph_rag 的主场（关系多跳）、又缺负样本。据此把 P2 重定为「**能力维度测试集先行，纵向对比与生成评测顺延**」。

**P2-1 指标 + 消融骨架 ✅ 已完成**
- `eval/metrics.py`：Recall@k / MRR / NDCG（纯函数，已单测）+ 跨策略命中的 `names_from_doc` 归一化。
- `eval/run_retrieval_eval.py`：6 路策略（vector/bm25/dual_level/hybrid/graph_rag/routed）隔离打分，出 CSV+缓存。
- 初步发现（--limit 10）：bm25 4ms/1.0、vector 0.95、**hybrid RRF 融合回到 1.0（证明 RRF 价值）**、dual_level 0.6、**graph_rag 与 routed=0 且 8.4s** → 暴露路由器把"怎么做"误路由给 graph_rag。

**P2-2 重构能力维度测试集 🔜 当前任务**
按"系统要回答的能力"而非"数据来源"组织，真值优先 Cypher 标注（可审计、可复现），规模 **120+ 条**：

| 能力 | 条数 | 示例 | 真值来源 |
|---|---|---|---|
| lookup 单菜 | ~20 | 宫保鸡丁怎么做 | Recipe.name |
| list 列表+过滤 | ~25 | 用鸡蛋做的低难度菜 | Cypher: REQUIRES + difficulty |
| **relation 关系多跳（graph 主场）** | ~40 | 和宫保鸡丁共用花生的菜 | Cypher 路径查询 |
| **negative 负样本** | ~20 | 怎么做满汉全席 | 真值=空，Cypher NOT EXISTS 校验 |
| reasoning 推理（少量·人工标） | ~15 | 番茄炒蛋和蛋汤哪个简单 | KG 属性 + 人工 |

- relation 做 4 个子型（**生成前先 Cypher 探测数据密度，保留可行的**）：共用食材 / 食材×分类组合 / 同分类避用某食材 / 按工具或做法。
- 每条 jsonl 字段：`id, query, capability, query_type, relevant_recipe_names, negative, label_source(cypher/human/llm_assisted), cypher, notes` —— 带 `capability` 可按能力切片出报告，带 `cypher` 任何标签都能复跑审计。
- 自然句改写层 → **后做**（核心跑通后再加，标签不变）。

**P2-3 纵向三系统对比 + 生成评测 ✅ 检索层纵向对比已完成（详见 问题与解决方案记录.md P2-3）**
- 同语料同题对比 C8/C9/当前：Recall@5 overall **C8 0.59 → C9 0.02（盲目加图谱+坏路由反而崩盘）→ 当前 0.84**；relation 0→0.95、MRR 0.05→0.86。生成评测（faithfulness/relevancy）此前 P2-3 已做。
- **纵向对比**：同一测试集跑 **C8（普通 RAG）/ C9（Graph 基线）/ 当前系统**，量化"从普通 RAG 到 Graph RAG 的进化收益"——简历主线故事。
- **生成评测**：LLM-as-judge 答案忠实度/相关性/完整度 + 拒答正确性（负样本防幻觉）。graph_rag 在此公平竞争（不再恒 0）。
- 保留 P2-1 的策略消融作为"为什么有效"的附录，不作主线。

**简历怎么写**（预期）：构建 120+ 条能力维度 golden 测试集（lookup/list/relation/negative/reasoning，真值 Cypher 可审计）；同题对比 C8 普通 RAG、C9 Graph 基线、当前系统，答案忠实度 +X%、关系类问题覆盖 Y%→Z%；用数据发现路由器对简单查询误用 graph_rag，按查询类型分流后召回 0→1、延迟降 95%。
**深挖预案**：golden 集标签怎么保证准（Cypher-grounded 可复跑）？为什么按能力维度而非数据来源？graph_rag 在召回层为 0 却为何保留（答案层价值）？负样本怎么测防幻觉？纵向对比为什么要同语料同题（控制变量）？

### ⚡ P1 Redis 多级缓存 ✅ 已完成（详见 问题与解决方案记录.md P1-1）
**目的**：后端工程深度的招牌，直接降延迟、省 LLM 钱，也是高并发故事的前置。

**状态**：4 级缓存（L1 进程 LRU / L2 Redis 精确 / L3 语义 BGE 余弦 / L4 路由决策 + embedding 缓存）全部实现，优雅降级 + invalidate + 埋点齐全。单测 20/20（FakeRedis）+ 真 Redis 烟测 8/8（Memurai 8.2.5）。完整基准：精确缓存 **p50 9834ms→0.0ms**，语义缓存 **p50 9834ms→385ms**，命中率 100%。

具体改动：新建 `code/cache/`
- **L1 精确缓存**：`query → 完整答案`，TTL 1h。命中直接返回。
- **L2 语义缓存**：query 向量化后，在 Redis 里找相似度 > 阈值（如 0.92）的历史 query，命中则复用其检索结果（甚至答案）。**这是最有讲头的一层。**
- **L3 embedding 缓存**：相同 query 的向量不重算。
- **L4 路由决策缓存**：相似 query 复用路由策略，省一次 LLM 路由调用。
- **失效策略**：知识库 rebuild 时 `FLUSHDB` 对应命名空间；TTL + 主动失效结合。
- **埋点**：记录命中率，喂给 P5 的指标。

**简历怎么写**：设计 4 级 Redis 缓存（精确/语义/embedding/路由），语义缓存基于 BGE 相似度匹配，重复 query 命中率 X%、p99 延迟从 Yms 降至 Zms、LLM 调用成本下降 W%。
**深挖预案**：语义缓存相似度阈值怎么定（太高不命中/太低答错）？缓存击穿/穿透/雪崩在这个场景怎么防？缓存和知识库更新的一致性怎么保证？为什么用 Redis 不用本地 LRU？

### 🌐 P3 FastAPI SSE 流式 + 前端可视化 ✅ 已完成（详见 问题与解决方案记录.md P3-1）
**目的**：从 CLI 变成能演示、能截图的产品。前端可视化让"检索过程"看得见，是面试最直观的加分项。

具体改动：
- **SSE 端点**：`GET /ask/stream`（Server-Sent Events），把 `generate_adaptive_answer_stream` 的 yield 通过 SSE 推给浏览器。先推一帧检索元数据（命中的菜谱、RRF 分数、路由策略、图谱子图），再流式推生成文本。
- **前端**（二选一，见决策点）：
  - 快版：Streamlit/Gradio（你挑战杯用过 Gradio，1–2 天出活）
  - 强版：Next.js + React（简历含金量更高，3–5 天）
- **可视化组件**：
  - 对话流（流式打字效果）
  - **检索溯源面板**：展示 top-k 文档 + 每路来源（向量/BM25/图）+ RRF 分数
  - **图谱可视化**：用 echarts/cytoscape 把图RAG 命中的 Neo4j 子图渲染出来（节点=菜谱/食材，边=关系）

**简历怎么写**：FastAPI SSE 实现流式问答；前端可视化检索溯源（RRF 多路分数）+ Neo4j 子图渲染，支持低延迟交互。
**深挖预案**：SSE vs WebSocket 怎么选（单向流式 vs 双向）？为什么先推检索元数据再推文本？图谱子图数据怎么从 Cypher 结果转成前端图结构？

### 🚀 P4 异步并发 + 限流 + 压测 ✅ 已完成（详见 问题与解决方案记录.md P4-1/2/3）
**目的**：拿到 QPS / p99 这些"生产级数字"，让高并发故事可信。这是你最想补的能力。

**状态**：async 端点 + asyncio.gather 三路并发 + AsyncOpenAI + Redis Lua 令牌桶限流 + asyncio/httpx 压测全做完。压测：repeat 场景 p50 **8ms**（缓存命中 80%）、unique 场景 p99 38s；瓶颈定位为生成 LLM（per-stage 平均 gate 0.5s/retrieve 4.0s/generate 19.6s）。

具体改动：
- **异步化**：把查询管道改 `async`。**关键优化**：三路检索（向量 / BM25 / 图KV）当前串行，用 `asyncio.gather` 并发执行，延迟近似降为最慢一路。
- **并发控制**：`asyncio.Semaphore` 限制 LLM 并发数（防 API 限流）；Neo4j driver / Milvus 连接池化。
- **限流**：Redis + Lua 实现令牌桶，保护 LLM API。
- **压测**：用 `locust` 或 `wrk` 压，出报告：QPS、p50/p95/p99 延迟、缓存开启/关闭对比、错误率。

**简历怎么写**：asyncio 重构查询管道，三路检索并发化使检索延迟下降 X%；Redis Lua 令牌桶限流；locust 压测单机 QPS Y、p99 Zms。
**深挖预案**：Python GIL 下 asyncio 为什么能提速（IO 等待期间切换）？三路并发用 gather 还是要不要限制并发数？令牌桶 vs 漏桶？压测时瓶颈在 LLM API 还是检索？背压怎么做？

### 📈 P5 可观测性 / LLMOps ✅ 已完成（轻量版，详见 问题与解决方案记录.md P5-1）
**目的**：能讲清"系统瓶颈在哪、怎么定位"。大厂 2026 面试趋势就是从"背八股"转向"线上排障"。

**状态**：Prometheus 指标（HTTP + 缓存命中/路由/拒答/LLM 调用/每步耗时 gate-retrieve-generate）+ `/metrics` 端点 + `/health` 附状态。已用它定位瓶颈=生成 LLM。Langfuse 全链路追踪留作后续可选增强。

具体改动：
- **Langfuse（开源，自部署）或 LangSmith**：接入后每一次 LLM 调用、每一路检索都有 trace，能在 UI 里看到完整调用链和每段耗时。
- **Prometheus 指标**（`/metrics` 端点）：request count、latency histogram、cache hit rate、route strategy 分布、各检索路召回数。
- **per-stage 延迟拆解**：路由 / 检索 / 生成各占多少 ms。

**简历怎么写**：Langfuse 全链路追踪 LLM 调用与检索；Prometheus 暴露 QPS/延迟/缓存命中率/路由分布指标，定位并优化了 XX 瓶颈。
**深挖预案**：一次查询里哪一步最耗时（通常是 LLM，但要用数据说）？怎么发现"某类 query 总是路由错"？trace 怎么串联多服务？

### 🐳 P6 异步任务队列 + 全栈部署 + CI
**目的**：交付能力 + 基础设施素养。也让你的"消息队列"学习有真实落地。

具体改动：
- **任务队列**（Celery + Redis 或 直接 Redis Stream）：把耗时任务异步化——知识库 rebuild、批量评测（P2）、你已有的数据构建 agent（`recipe_ai_agent.py`）。用户提交 → 入队 → Worker 处理 → 完成回调。
- **全栈 docker-compose**：一个文件拉起 app + Neo4j + Milvus + Redis +（Langfuse），`docker-compose up` 一键起整个系统。
- **CI**（GitHub Actions）：lint + 单测 + P2 评测脚本；指标退化阻断合并。

**简历怎么写**：Celery+Redis 异步化知识库重建与批量评测；docker-compose 一键部署全栈（app/Neo4j/Milvus/Redis）；GitHub Actions CI 守住检索指标不退化。
**深挖预案**：重建知识库为什么要走 MQ（耗时长、不能阻塞请求）？消息丢了怎么办（持久化+ack）？消费失败的重试与死信队列？
