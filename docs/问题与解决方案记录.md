## 环境与工具（ENV）

### ENV-1 `conda run` 不支持带换行的 `-c` 脚本
- **现象**：`conda run -n graph-rag python -c "<多行脚本>"` 报
  `AssertionError: Support for scripts where arguments contain newlines not implemented.`
- **根因**：`conda run` 已知限制，不处理含换行的命令行参数。
- **解决**：直接用 conda 环境的 python 全路径执行，单行脚本或脚本文件：
  - 环境路径：`E:/APPS/Anaconda3/envs/graph-rag/python.exe`
  - 多行逻辑写成 `.py` 文件再用上述 python 跑；或写成单行（分号/列表推导）。
- **涉及文件**：无（环境操作约定）。

### ENV-2 Docker Desktop 未启动 / 数据库容器停止
- **现象**：`docker ps` 报 `failed to connect to the docker API ... dockerDesktopLinuxEngine`；`neo4j-db`、`milvus-*` 容器状态为 `Exited (2 weeks ago)`。
- **解决**：
  1. 启动 Docker Desktop：`cmd /c start "" "D:\APPS\Docker\Docker\Docker Desktop.exe"`
  2. 轮询等待守护进程就绪（`docker ps` 成功为止）。
  3. 容器已存在、配置未变，直接 `docker start milvus-etcd milvus-minio milvus-standalone neo4j-db`（**无需重建**）。
  4. 轮询 `docker inspect --format '{{.State.Health.Status}}'` 直到 `neo4j-db` 与 `milvus-standalone` 均 `healthy` 再起服务。
- **注意**：`neo4j-init` 是一次性数据导入容器，`Exited (0)` 是正常终态，无需重启。

## P0 工程化重构

### P0-1 LLM 提供商配置混乱（MOONSHOT vs 智谱 vs Kimi）
- **现象**：`code/rag_modules/generation_integration.py` 同时存在：
  - 读环境变量 `MOONSHOT_API_KEY`
  - `base_url` 硬编码为 `https://open.bigmodel.cn/api/paas/v4`（**智谱**）
  - 默认 `model_name="kimi-k2-0711-preview"`（**Kimi**）
  - 而 `code/.env` 的 `MOONSHOT_API_KEY` 值实为**智谱 key**、`config.py` 的 `llm_model="glm-4-flash"`。
  三方不一致，命名误导，难维护。
- **根因**：配置分散在 `.env` / `config.py` / `config.json` / 模块硬编码，没有单一真相源。
- **解决**（配置收口）：
  - `config.py` 新增字段 `llm_api_key`、`llm_base_url` + `GraphRAGConfig.from_env()`，默认统一**智谱 GLM**。
  - 密钥解析链：`LLM_API_KEY → ZHIPU_API_KEY → MOONSHOT_API_KEY`（向后兼容现有 .env，不破坏可用性）。
  - `generation_integration.py` 改为从 config/参数/环境变量注入，去掉硬编码。
  - `main.py` 把 `self.config.llm_api_key/llm_base_url` 注入生成模块。
- **涉及文件**：`code/config.py`、`code/rag_modules/generation_integration.py`、`code/main.py`、`code/.env.example`。
- **验证**：`from config import DEFAULT_CONFIG` → `llm_model=glm-4-flash`、`llm_base_url=bigmodel.cn`、key 成功加载、`to_dict()` 脱敏为 `***`。
- **面试可讲点**：多源配置收口、向后兼容的密钥解析链、敏感信息脱敏。

### P0-3 Windows 控制台 GBK 编码崩溃 ⭐
- **现象**：服务启动到 `initialize_system()` 的 `print("✅ 高级图RAG系统初始化完成！")` 时抛
  `UnicodeEncodeError: 'gbk' codec can't encode character '✅'`，启动失败（exit 3）。
- **根因**：Windows 中文系统控制台默认编码 GBK，无法编码 emoji（✅/❓/📊 等大量存在于 `main.py` 的 `print`）；当 stdout 被重定向到文件时，Python 仍按 locale(GBK) 编码流。
- **解决**：在入口处强制 stdout/stderr 为 UTF-8：
  ```python
  try:
      sys.stdout.reconfigure(encoding="utf-8")
      sys.stderr.reconfigure(encoding="utf-8")
  except Exception:
      pass
  ```
  加在 `api/server.py` 与 `main.py` 顶部（在任何 print 之前）。
- **涉及文件**：`code/api/server.py`、`code/main.py`。
- **验证**：重启后 `✅ 检索引擎初始化完成！` 正常输出，`Application startup complete`。
- **面试可讲点**：Python 流编码机制、Windows locale 编码坑、`PYTHONUTF8`/`reconfigure` 方案对比。

### P0-4 Windows 下 curl 传中文 body 被损坏
- **现象**：`curl -d '{"question":"番茄炒蛋..."}'` 请求 `/api/ask` 返回 HTTP 400
  `{"detail":"There was an error parsing the body"}`；服务端日志显示 `POST /api/ask 400`。
- **根因**：Git Bash 的 curl 在传递含中文的命令行参数时，UTF-8 字节被破坏，FastAPI JSON 解析失败。（服务端本身正常——`/health` 200、`/api/ask` 用英文 body 可过。）
- **解决**：测试改用 Python `requests`，`json=` 参数自动以 UTF-8 + 正确 `Content-Type` 发送 → 写成 `code/_smoke_ask.py` 复用。
- **涉及文件**：`code/_smoke_ask.py`（新增冒烟测试）。
- **面试可讲点**：HTTP 请求体编码、Content-Type、跨平台测试工具选择。

### P0-5 contextvars + StreamingResponse 跨上下文崩溃 ⭐⭐（最有讲头）
- **现象**：`/api/ask/stream`（SSE）流式输出**本身成功**（客户端已收到 metadata、token、done 帧），但流结束后服务端抛
  `ValueError: <Token ...> was created in a different Context`，连接被异常切断，客户端报 `ChunkedEncodingError: Response ended prematurely`。
- **根因**（contextvars + 线程池的经典坑）：
  - `StreamingResponse` 接收的同步生成器，Starlette 通过 `iterate_in_threadpool` 把**每个 `next()`** 分别派发到 worker 线程（`anyio.to_thread.run_sync`），每个 worker 线程**复制一份当前异步上下文**。
  - 原实现用 `with RequestContext()`：`__enter__` 在 worker 线程 A 的上下文里创建 token，`__exit__`（生成器结束时）在另一个 worker 线程的上下文里跑 → 上下文不一致 → `contextvars.Token.reset` 报错。
  - 对比：`/api/ask`（非流式）整个 handler 跑在**同一个** worker 线程，enter/exit 同上下文，故不崩。
- **解决**：改用 **HTTP 中间件**在请求级异步上下文里注入 request_id：
  ```python
  @app.middleware("http")
  async def request_id_middleware(request, call_next):
      rid = request.headers.get("x-request-id") or uuid.uuid4().hex[:12]
      request_id_var.set(rid)            # 写进请求级异步上下文
      response = await call_next(request)
      response.headers["x-request-id"] = rid
      return response
  ```
  - worker 线程复制异步上下文 → 自动继承 rid；
  - **故意不 reset**：请求结束该异步上下文整体销毁，不泄漏到下一个请求（reset 反而跨 Context 报错）。
  - handler 内改为 `rid = get_request_id()` 读取，不再用 `with`。
- **涉及文件**：`code/api/server.py`、`code/core/logging_config.py`。
- **验证**：`_smoke_stream.py` → `metadata:1 / token:102 / done:1 / error:0`，断言全过，连接正常关闭。
- **面试可讲点**（高级）：contextvars 传播机制、asyncio 任务/上下文生命周期、Starlette 同步生成器在线程池的执行模型、为什么中间件 set 不 reset 是安全的、分布式链路追踪 request_id 的正确注入姿势。

## P2 评测体系

### P2-1 graph_rag 在召回指标上恒为 0 → 评测方法论重构 ⭐⭐（最有讲头）
- **现象**：P2 初版用「系统内 6 路策略横向消融」评测，在 golden 测试集（前 10 条 single_recipe "怎么做"）上实测：

  | 策略 | Recall@5 | MRR | 平均延迟 |
  |---|---|---|---|
  | vector | 1.000 | 0.950 | 527ms |
  | bm25 | 1.000 | 1.000 | **4ms** |
  | hybrid（RRF 融合） | 1.000 | 1.000 | 2134ms |
  | dual_level（图KV） | 0.600 | 0.600 | 2636ms |
  | **graph_rag** | **0.000** | **0.000** | 2954ms |
  | **routed（线上路由）** | **0.000** | **0.000** | **8442ms** |

  graph_rag 与线上 routed 在 Recall/MRR/NDCG 上**恒为 0**。
- **根因（双重，必须分开看）**：
  1. **度量错位（对 graph_rag 不公平）**：graph_rag 返回的是**子图/路径描述**，`_subgraph_to_documents` 把 `recipe_name` 写死成"知识子图"、正文也不含具体菜谱名；而指标按"菜谱名召回"计算 → graph_rag 结构上不可能命中 → 必然 0。相当于**用爬树能力评鱼**：0 分不能说明鱼不行，只能说明量错了维度。
  2. **测试集偏斜 + 路由缺陷（却是真问题）**：测试集 63% 是"X怎么做"单菜查询（relevant=1，Recall 退化为二元），既缺 graph_rag 的主场（关系多跳），又缺负样本；且路由器把这类简单查菜查询**误路由给 graph_rag**（慢、召回 0），routed 继承了 graph_rag 的 0 并叠加 8.4s 延迟。
- **解决（评测方法论重构）**：
  1. 把 P2 从「纯横向消融」重定为「**能力维度测试集先行**」：按 lookup / list / **relation** / **negative** / reasoning 配比；relation 类用 Cypher 路径查询标真值（graph_rag 公平上场），negative 类（真值=空）改测**拒答/防幻觉**。
  2. graph_rag 的真正价值改由「**生成阶段 LLM-as-judge**」（答案忠实度/相关性）评测，不再用召回指标评它 → 顺延到 P2-3。
  3. 新增「**纵向三系统对比**」（C8 普通 RAG / C9 Graph 基线 / 当前，同语料同题）作为简历主线 → 顺延到 P2-3。
  4. 保留原消融作为「为什么有效」的附录，不作主线。
- **涉及文件**：`eval/metrics.py`、`eval/run_retrieval_eval.py`（已建）；`项目优化路线图.md`（P2 段重写为 P2-1/P2-2/P2-3）；将写 `eval/build_testset_v2.py` + `eval/testset.v2.jsonl`。
- **验证**：--limit 10 实测表如上；路线图 P2 已按新方法论更新。
- **面试可讲点**（高级评测思维）：① **度量对齐**（measuring the right thing）——产物不同就该用不同指标；② 按**能力维度**而非数据来源设计测试集；③ 负样本测防幻觉、关系类测图推理；④ 用评测发现并否决"花哨模块的滥用"（路由器误用 graph_rag）；⑤ 纵向对比的**控制变量**（同语料同题）。"我用数据发现评测不公平，于是重构了评测方法论"本身就是高级信号。

## P2 评测体系（续）

### P2-2 graph_rag 返回空子图 → 五层定位与修复 ⭐⭐⭐（迄今最大排查）
- **现象**：v2 评测 graph_rag 在所有能力 Recall=0（含其"主场" relation）。诊断 `graph_rag_search` 返回 **0 节点空子图**（正文 literally `"关于  的知识网络，包含 0 个相关概念和 0 个关系。"`，中心实体名都为空）。更隐蔽的是：之前 API 冒烟"番茄炒蛋怎么做"看似答得不错——其实是 **graph_rag 给了空上下文、LLM 用参数知识硬编**，伪装成 RAG 成功。评测+诊断才戳破。
- **根因（5 层，逐层定位，都有实锤）**：
  1. 子图 Cypher `WHERE size(neighbors) <= $max_nodes` 把 >50 邻居的子图**整块丢弃** → 空。（实测：炸鲜奶 depth2=359 邻居被丢；depth1=25 本可保留。）
  2. LLM 把**类型词** `'Recipe'` 当 source_entities，真正判别实体（胡椒粉）被放进 target_entities；而子图提取**只认 source** → 从 `'Recipe'` 这个 ConceptType 节点（连着全图 5783 邻居）展开 → 被丢。
  3. 子图提取完全**忽略 target_entities**。
  4. `_build_subgraph_description` 只写"包含 N 个相关概念"，**不列邻居菜谱名** → 即使子图非空，下游菜谱名匹配也命中不了。
  5. 邻居截断**无优先级**，depth-2 噪声（步骤/食材/schema 节点）淹没菜谱节点。
- **修复（4 处，`rag_modules/graph_rag_retrieval.py`）**：
  - **B**：删 `WHERE size<=max_nodes`，改**截断** `collect(DISTINCT neighbor)[0..max_nodes]`。
  - **A**：种子只用真实数据节点 `source.nodeId >= '200000000'`（过滤 ConceptType/Category）；子图分支把 `target_entities` 并入 source 种子。
  - **C**：`_build_subgraph_description` 列出邻居实体名（top30）。
  - **D**：邻居按 **`Recipe 优先 + 度数 DESC`** 排序再截断（答案型节点排前）。
- **验证**（子集 `--per-cap 6`，graph_rag Recall@5 修复前后）：

  | 能力 | 修前 | 修后 |
  |---|---|---|
  | lookup | 0.000 | **1.000** |
  | list | 0.000 | 0.208 |
  | reasoning | 0.000 | 0.167 |
  | relation | 0.000 | 0.000（仍 0，见下） |

  `routed`（继承 graph_rag）lookup 同步 0→1.0。
- **残留设计限制（不是 bug，已识别不再硬修）**：graph_rag 在 relation 仍 0。根因 = 通用 depth-N 子图**无法施加查询所需的分类/关系过滤**（"用了胡椒粉的主食" 要的是 `主食` 类，子图返回的是"白胡椒粉的鱼虾聚类"）。要真正赢 relation，须把关系查询编译成**目标 Cypher**（带分类/关系过滤），而非给通用子图打补丁。
- **决策**：① 修路由器——relation 查询走 hybrid（实测 relation Recall@5=0.48 最强），别给 graph_rag；② graph_rag 的剩余价值（给 LLM 的探索上下文）交 **P2-3 生成评测**判断。
- **涉及文件**：`rag_modules/graph_rag_retrieval.py`（`extract_knowledge_subgraph` / `graph_rag_search` 子图分支 / `_build_subgraph_description`）。
- **面试可讲点**（完整工程闭环）：「评测发现模块**静默失效**（被 LLM 硬编能力掩盖）→ 逐层定位 5 个根因 → 修复后 lookup 召回 **0→100%** → 但**诚实识别出通用子图的结构化查询精度上限**，据此调整路由器而非硬凑。」即"评测→定位→修复→验证→知止"。

### P2-3b 拒答机制防幻觉（向量闸门 + 提示词拒答）⭐
- **现象**：P2-3 生成评测发现 Refusal=0——负样本（北京烤鸭等）100% 被编造答案，系统从不拒答（呼应"番茄炒蛋看似答对实则 LLM 硬编"）。
- **探测定标**：向量相似度——存在菜谱 0.6–0.8、明显无关 0.4 左右；但**语义相近菜名（北京烤鸭→啤酒鸭）分数 0.73 比真菜还高**，纯阈值挡不住。
- **机制（双层）**：
  - 层3 向量闸门 `check_answerable`（score<0.45 直拒、省一次生成）：挡明显无关（Python/天气）。
  - 层1 提示词拒答（生成 prompt 强制）：让 LLM 据上下文判"菜名对不上/完全无关"则拒。
- **校准迭代**：初版提示词太激进 → 正样本误拒 41.7%（列表/关系查询被误伤）；收窄规则（列表/关系查询不全也作答）→ 误拒 0%。
- **验证**（20 负 + 12 正）：

  | 指标 | 修前 | 初版 | 校准后 |
  |---|---|---|---|
  | 负样本拒答率 | 0% | 80% | **70%** |
  | 正样本误拒率 | 0% | 41.7% | **0%** |

- **残留（已知限制）**：剩 30% 泄漏全是**语义相近菜名**（松鼠鳜鱼/剁椒鱼头/酸菜鱼/辣子鸡/香辣虾/红烧肉不放酱油）——KB 有同类真菜（鱼/鸡/虾/红烧肉），LLM 见相关上下文就编。需 entity/同义词处理（见 `改进待办清单.md`）。
- **涉及文件**：`rag_modules/generation_integration.py`（拒答提示词）、`main.py`（`check_answerable`+`REFUSAL_MSG`+ask 闸门）、`api/server.py`（`/api/ask` 与 `/ask/stream` 闸门）、`eval/run_refusal_eval.py`（验证）。
- **面试可讲点**："评测发现系统对不存在问题 100% 幻觉作答；设计向量闸门+提示词双层拒答；负样本集回归发现初版误伤正常查询(41.7%)，收窄后拒答率 0→70%、正样本 0 误拒；并识别出语义相近菜名是剩余硬 case。"——发现→设计→校准→验证→知限。

### P2-2c 路由器误路由 graph_rag → 数据支撑改为默认 hybrid（C）
- **现象**：`routed`（线上路由）在 P2-2 评测 relation/reasoning 仍弱，且 reasoning 出现 **108s 长尾**。根因：LLM 路由器把绝大多数查询推荐给 graph_rag——cooking 查询的 `relationship_intensity` 普遍被打到 0.8+，任何阈值都难以稳定区分。
- **修复**：`intelligent_query_router.py` 新增 `_routing_policy`，用 P2 评测结论**覆盖 LLM 推荐**——当前一律 `hybrid_traditional`（数据：hybrid 在所有能力最强或并列；graph_rag 无任何能力超过 hybrid）。LLM 分析保留，用于 `explain_routing_decision` 与 metadata。
- **验证**：4 类查询（lookup/list/relation/reasoning）路由后均为 `hybrid_traditional`，结果正确（"用了胡椒粉的主食" top3 含 `热干面` 命中 relevant；"宫保鸡丁怎么做" top1=`宫保鸡丁`）。
- **后续**：graph_rag 的重新启用留待 [B] 目标 Cypher 改造或 P2-3 生成评测验证；"按查询能力细化分流"见 `改进待办清单.md`。
- **涉及文件**：`rag_modules/intelligent_query_router.py`。
- **面试可讲点**：用评测数据否决了"看起来很酷实则更差"的 LLM 路由选择——以指标为准、不为花哨买单。

### P2-[B] graph_rag 关系查询 → 目标 Cypher 编译器（relation 召回 0→0.91 翻盘）⭐⭐⭐（graph_rag 救赎）
- **背景**：P2-2/P2-2c 的判决是"graph_rag 通用子图在 relation 无分类/工具过滤 → 召回 0，已禁用，路由切 hybrid"。改进待办 [B] 是让 graph_rag 真正翻盘的唯一出路——把关系查询编译成**带过滤的精确 Cypher**（与测试集真值同构），而非通用 depth-N 子图。
- **实现**（`rag_modules/graph_rag_retrieval.py` + `intelligent_query_router.py`）：
  1. **正则识别 4 子型 + 抽槽位**（`detect_relation_pattern`，顺序敏感）：`shared_ingredient`(和X一样用了Y的菜) / `ingredient_category`(用了X的Y) / `by_tool`(需要T的菜) / `by_method`(用到M这种做法的菜)。纯规则，不依赖 LLM。
  2. **图内校验实体名**（`_resolve_name`）：精确匹配优先，其次取最短 CONTAINS 命中——顺带修了【种子歧义】（`胡椒粉` 不再落到 `白胡椒粉`）。
  3. **编译精确 Cypher**（`_compile_relation_cypher`，参数化防注入）：每子型一条带过滤的 Cypher。
  4. **富化菜谱文档**（`_enrich_recipes`）：命中菜名补全食材/分类/难度，供下游生成。
  5. **接入**：`graph_rag_search` 关系查询优先走 `relational_search`（命中即返回，跳过通用子图）；路由器 `_routing_policy` 按关系模式分流到 graph_rag，graph_rag 关系未命中时降级 hybrid。
- **关键根因（排查 shared_ingredient 全 0）**：探针发现图谱**食材节点按菜谱重复创建**——"鸡翅中"有 3 个同名 Ingredient 节点，可乐鸡翅与烤鸡翅各指向不同节点。最初照抄 testset 真值 Cypher 的 `<-[:REQUIRES]-(r)`（要求两菜共享*同一*节点）必然 0 命中。改为**按食材名 join**（与 `gen_relation` 标签生成同构）后 shared_ingredient 召回 0→0.972。注：testset 里那条真值 Cypher 本身也因此是"脆的"（依赖节点身份），但标签是对的（用名字 join 算的）。
- **验证（全量 v2 评测，relation n=37）**：

  | 策略 | Rec@1 | Rec@3 | Rec@5 | MRR | NDCG@5 | avg_ms |
  |---|---|---|---|---|---|---|
  | hybrid | 0.101 | 0.347 | 0.449 | 0.487 | 0.420 | 2096 |
  | **graph_rag** | 0.356 | 0.820 | **0.909** | **1.000** | **0.976** | **12** |
  | **routed** | 0.356 | 0.820 | **0.909** | **1.000** | **0.976** | 3882 |

  - graph_rag relation **Rec@5 0→0.909（hybrid 的 2 倍）**，MRR 完美 1.000（返回的全是相关菜，top1 必中），延迟 12ms（纯 Cypher，比 hybrid 快 ~170×）。
  - **routed 现在每一项 ≥ hybrid**：lookup(1.0=1.0)/list(0.304=0.304)/relation(0.909 ≫ 0.449)/reasoning(1.0=1.0) → **routed 成为全局最优策略**（修前 routed 因误路由 graph_rag 全弱）。
  - 无回退：graph_rag lookup 仍 1.000（P2-2 子图修复保持；relational_search 不影响 lookup）。
  - by_tool 的 Rec@5=0.69（Precision 1.0）miss 全因 gold>5 被 top_k=5 截断（hybrid 同等约束）。
- **诚实的边界**：graph_rag list(0.174)/reasoning(0.533) 仍弱于 hybrid——但路由器正确地让这些走 hybrid，所以 routed 整体最优。负样本误检索率 graph_rag 仍 100%（返回子图），由 P2-3b 拒答闸门在生成层兜底，非回退。
- **涉及文件**：`rag_modules/graph_rag_retrieval.py`、`rag_modules/intelligent_query_router.py`、`eval/_diag_relation.py`、`eval/_probe_shared.py`。
- **面试可讲点（最强故事）**：① 不靠"看起来很酷"的通用图遍历，而是诊断出其结构性短板（无法施加属性过滤）→ 用**目标 Cypher 编译**针对性补救；② 排查"照抄真值 Cypher 却全 0"时，探针发现图谱**食材节点按菜谱重复**这一隐藏数据特征，据此改 join 策略——体现"先理解数据再写查询"；③ 用评测数据驱动路由（关系→graph_rag，其余→hybrid），让 routed 成为全局最优而非拍脑袋；④ 延迟从 2s 降到 12ms（去 LLM 化）。完整弧线：发现短板→定位根因→精准修复→评测验证翻盘→路由整合。

## P1 Redis 多级缓存

### P1-1 多级缓存设计与优雅降级 ⭐⭐
- **目的**：降延迟 + 省 LLM 钱，高并发故事前置。hybrid/routed 单次 2–19s（检索 + LLM 关键词抽取 + 生成），重复/相似 query 应瞬返。
- **设计**（新建 `code/cache/` 包，命名空间前缀 `grag:`）：

  | 层 | 缓存内容 | 键 | TTL |
  |---|---|---|---|
  | L1 进程内 LRU | 答案 | normalized query | 进程内（容量 512） |
  | L2 Redis 精确 | 答案+溯源+策略 | `grag:answer:<sha1>` | 1h |
  | L3 语义 | 相似 query 复用答案（BGE 余弦 ≥ 阈值） | `grag:sem:*` hash | 1h |
  | L4 路由决策 | analyze_query 结果（跳过 LLM 路由） | `grag:route:<sha1>` | 6h |
  | (+) embedding | query 向量（语义缓存不重算） | `grag:emb:<sha1>` | 24h |

- **优雅降级（核心）**：`RedisClient` 探活失败 → `available=False` → `CacheManager` 所有 `get_*` 返回 None（miss）、`set_*` no-op，系统行为与"无缓存"完全一致。运行中 Redis 掉线也 try/except 兜底不抛业务层。
- **线程安全**：FastAPI 同步端点跑线程池 → L1/metrics/语义索引用 `RLock` 保护；Redis 客户端自带连接池。
- **失效**：知识库 rebuild 调 `invalidate_all()`（SCAN 删 `grag:*` + 清 L1/语义镜像）；TTL 兜底。API 暴露 `POST /api/cache/invalidate`。
- **关键优化**：精确缓存(L1/L2)检查置于**拒答闸门之前**——缓存里的答案都已过闸门，命中时重跑闸门（~300ms 向量查）是纯浪费；语义缓存(L3)留闸门之后更安全（避免误命中本该拒答的近义 query）。
- **验证**：
  - 单测 `eval/_test_cache.py`（FakeRedis 后端）**20/20 通过**：L1 LRU 淘汰、L2 精确、L3 语义余弦、L4 路由、embedding 缓存、归一化、降级、invalidate、stats。
  - 真 Redis 烟测 `eval/_smoke_live_redis.py` **8/8 通过**（Memurai 8.2.5）：L2 精确 / 归一化等价 / L4 路由 / L3 语义命中 / L3 语义不命中 / embedding 缓存 / invalidate。
  - 降级端到端：Redis 未起时系统照常问答（关系查询走 [B] 答对，10.1s 全链路）。
  - **完整基准 `eval/bench_cache.py`（Memurai 8.2.5 真实数字）**：

    | 档 | 命中率 | p50 | p99 | 命中分布 |
    |---|---|---|---|---|
    | COLD（冷启动，全链路） | 0% | 9834ms | 43631ms | MISS×7 |
    | WARM（精确 L1/L2） | 100% | **0.0ms** | **0.0ms** | L1×7 |
    | SEMANTIC（L3 语义） | 100% | 385ms | 411ms | L3×3 |

    - 精确缓存：**p50 9834ms → 0.0ms**（命中跳过闸门+检索+生成，瞬返）。
    - 语义缓存：**p50 9834ms → 385ms**（同义不同字面 query 复用答案；385ms 是拒答闸门的 Milvus 向量查开销，因语义缓存放闸门之后）。
    - 测试集跨 lookup/list/relation/reasoning 四能力 + 3 组改写对（"怎么做"↔"如何做"↔"的做法"、"用鸡蛋做的菜有哪些"↔"哪些菜用鸡蛋做的"）全部命中。
- **本机 Redis 部署小坑**：docker 拉 `redis:7-alpine` 经 docker.1ms.run/xuanyuan/daocloud 三源均超时（网络问题）；排查发现 6379 已被一个**老 Windows Redis 服务（不支持 HELLO/RESP3）占用**→ `sc stop/delete Redis`（管理员）清掉 → 装 **Memurai 8.2.5**（Windows 原生 redis）解决。
- **涉及文件**：`cache/redis_client.py`、`cache/cache_manager.py`、`cache/__init__.py`、`config.py`、`main.py`、`rag_modules/intelligent_query_router.py`、`api/server.py`、`data/docker-compose.yml`（加 redis 服务）、`eval/_test_cache.py`、`eval/bench_cache.py`、`eval/_smoke_live_redis.py`。
- **面试可讲点**：① 为什么多级（L1 防 Redis 往返、L2 跨进程持久、L3 同义复用、L4 省路由 LLM）；② 语义缓存阈值怎么定（0.92：太高不命中/太低答错，BGE 余弦经验区间）+ 误命中怎么防（留闸门后）；③ 缓存击穿/穿透/雪崩在此场景怎么防（TTL + 单飞可加；负样本不缓存防穿透）；④ 缓存与 KB 一致性（rebuild 主动 invalidate + TTL 兜底）；⑤ 为什么精确缓存放闸门前（缓存项已过闸门，省 300ms）；⑥ 优雅降级——Redis 是加速器不是依赖。

## P5 可观测性 + P4 异步并发/限流/压测

### P5-1 Prometheus 指标 + 每步耗时（"线上哪慢一眼看到"）⭐⭐
- **目的**：拿到"生产级数字"+ 能定位瓶颈。新建 `core/metrics.py`，FastAPI middleware 自动记 HTTP，业务层手动记。
- **指标**（命名空间 `graphrag_`）：
  - HTTP：`http_requests_total{endpoint,method,status}`、`http_request_duration_seconds` 直方图。
  - 业务：`cache_hits_total{layer}`（L1/L2/L3/MISS）、`route_strategy_total{strategy}`、`refusals_total{reason}`、`llm_calls_total{stage}`、`stage_duration_seconds{stage}`（gate/retrieve/generate 每步耗时直方图）。
  - `/metrics` 端点供 Prometheus 抓取；`/health` 附带缓存+限流状态。
- **关键发现（P5 的价值）**：压测 43 个全链路请求的每步平均耗时——**gate 0.5s / retrieve 4.0s / generate 19.6s**。**瓶颈是生成（LLM）而非检索**，单请求里生成占 ~80%。这直接否决了"优化检索提吞吐"的方向，把火力指向生成（缓存已让重复瞬返；异步让多请求并发等 LLM）。

### P4-1 全异步重构（async 端点 + 三路并发 + AsyncOpenAI）⭐⭐
- **设计**：API 端点改 `async def`；阻塞调用（Milvus 闸门、analyze LLM、生成 LLM）全部让出事件循环——同步实现用 `asyncio.to_thread` 卸到线程池，生成另迁 **AsyncOpenAI**（最耗时的 I/O 真正 yield）。
- **三路并发**：`HybridRetrievalModule.hybrid_search_async` 用 `asyncio.gather` 并发跑 dual_level/vector/bm25，延迟从三者之和 ≈ 最慢一路。
- **同步 CLI 路径保留不动**（`ask_question_with_routing` 仍是同步，CLI 用）；新增 `ask_async`（API 用）。
- **涉及文件**：`rag_modules/{generation_integration,hybrid_retrieval,graph_rag_retrieval,intelligent_query_router}.py`（各加 async 方法）、`main.py`（`ask_async`）、`api/server.py`（async 端点）。
- **诚实的边界**：未把 neo4j/milvus 客户端改成原生 async（风险高、收益已被 to_thread 覆盖）； AsyncOpenAI 是收益最大、风险最低的一步，已做。面试可讲：为什么用 to_thread 而非全改 async 客户端（工程权衡：sync 库成熟稳定，to_thread 拿到并发收益且可回退）。

### P4-2 Redis 令牌桶限流（Lua 原子）⭐
- **设计**：`core/rate_limiter.py`，按 IP 限流，**Lua 脚本在 Redis 内原子完成"补充令牌→判断→扣减"**（避免并发竞态）。Redis 挂时降级进程内令牌桶（带锁）。FastAPI middleware 对 `/api/*` 生效，超额返回 429。
- **验证**（真 Redis）：容量 3、补充 2/s → 突发 5 个放行 3 拒绝 2；等 1.1s 后再 3 个放行 2（令牌已补充）。✓
- **可调**：`RATE_LIMIT_CAPACITY`（突发上限）、`RATE_LIMIT_REFILL`（持续 QPS）。

### P4-3 压测（asyncio+httpx）⭐⭐
- **脚本** `eval/load_test.py`：N 并发压 `/api/ask`，报 QPS/p50/p95/p99/错误率/缓存命中分布。两场景对比：repeat（小池反复问，缓存友好）vs unique（大池少重复，压真实链路）。
- **结果**（10 并发，限流临时关闭以测真实吞吐）：

  | 场景 | 请求 | QPS | p50 | p99 | 缓存命中 |
  |---|---|---|---|---|---|
  | repeat | 60 | 1.63 | **8ms** | 29.4s | 48/60 (80%) |
  | unique | 30 | 0.68 | 9ms* | 38.5s | 17/30 |

  - \* unique 的 p50=9ms 是因部分 query 与 repeat 池重叠命中缓存；纯 miss 的 p95/p99=22–38s。
  - **缓存价值**：重复查询 p50 从 ~10s 降至 **8ms**（命中 L1）；冷 miss 走全链路受生成 LLM 延迟主导（平均 19.6s，方差大）。
  - **异步价值**：10 个请求能同时 in-flight 各等自己的 LLM 调用；同步端点下每个会占满一个线程池线程 ~20s。
- **瓶颈结论**：吞吐上限由生成 LLM 决定（非检索）。生产改进方向=更稳/更快的 LLM 或流式首字；本项目层面，缓存（重复瞬返）+ 异步（并发等 LLM）已是当前最优解。

- **涉及文件**：`core/{metrics,rate_limiter}.py`、`api/server.py`、`eval/load_test.py`、`config.py`（限流参数）。
- **面试可讲点**：① 用 Prometheus per-stage 耗时定位瓶颈在生成而非检索（数据否决直觉）；② 令牌桶 vs 漏桶（突发友好）、为什么 Lua（原子）；③ 为什么 to_thread 而非全 async 客户端（工程权衡）；④ Python GIL 下 async 为什么能提吞吐（I/O 等待时事件循环切其它请求）；⑤ 缓存+异步是应对 LLM 瓶颈的组合拳。

### P2-3 纵向三系统对比（C8 普通 RAG → C9 Graph 基线 → 当前）⭐⭐⭐（简历主线故事）
- **目的**：同语料、同题、同库对比三个系统，量化"从普通 RAG 到 Graph RAG"的进化收益——简历主线。
- **方法**：同一份 golden 测试集（testset.v2.jsonl，Cypher 真值），三个系统各跑检索、提菜谱名、同一套指标（Recall/MRR/NDCG）。公平性：三者菜谱同源（C8 FAISS 322 道覆盖 gold 98%；C9 与当前共用同一 Neo4j+Milvus）。C8/C9 与当前模块同名，**各自独立子进程**跑（`eval/run_c8_baseline.py`、`run_c9_baseline.py`）避免 import 串台。C9 慢（graph_rag 每条调 LLM），取 32 条子样本（每能力 8）做三方公共集。
- **结果（Recall@5 / MRR）**：

  | 系统 | lookup | list | relation | reasoning | **overall R@5** | MRR |
  |---|---|---|---|---|---|---|
  | C8 普通 RAG（向量） | 1.00 | 0.35 | 0.26 | 0.75 | **0.59** | 0.59 |
  | C9 Graph 基线（graph_rag） | 0.00 | 0.04 | 0.00 | 0.00 | 0.01 | 0.03 |
  | C9 Graph 基线（routed，端到端） | 0.00 | 0.08 | 0.00 | 0.00 | **0.02** | 0.05 |
  | 当前（graph_rag） | 1.00 | 0.20 | **0.95** | 0.38 | 0.63 | 0.72 |
  | **当前（routed，端到端）** | 1.00 | 0.41 | **0.95** | **1.00** | **0.84** | **0.86** |

- **三个关键结论**：
  1. **盲目加图谱反而拖垮系统**：C9（0.02）比什么都不加的 C8（0.59）**差 30 倍**。根因有二：① C9 的 graph_rag 用"通用 depth-N 子图 + LLM 抽实体"，中文长句查不到节点 → 关系召回 0；② C9 路由器把几乎所有查询误丢给 graph_rag（C9-routed ≈ C9-graph_rag ≈ 0 印证）。lookup 都归零——一个连单菜查不出的"智能"系统。
  2. **修好图谱才真有用**：当前系统用 [B] 目标 Cypher 修好 graph_rag（relation 0→0.95）+ P2-2c 数据驱动路由（关系→graph_rag，其余→hybrid），overall 0.02→**0.84**，relation/reasoning 接近满分。
  3. **路由是放大器**：C9 的坏路由把还能用的 hybrid 也埋了（routed≈graph_rag）；当前的好路由让每条查询走各自最优策略，routed 全局最优。
- **诚实交代**：C8 是纯向量（vanilla RAG 典范），未用其 hybrid（避免依赖 C8 数据目录重建 BM25）；C9 端到端即其出厂路由行为（含 bug，正是对比对象）；32 条子样本因 C9 慢，但每能力 8 条已足以呈现量级差异。
- **涉及文件**：`eval/run_c8_baseline.py`、`eval/run_c9_baseline.py`、`eval/compare_longitudinal.py`、输出 `eval/results/{long_c8,long_c9}.jsonl` + `longitudinal_compare.csv`。
- **简历可写**：*"同语料同题纵向对比 C8 普通 RAG、C9 Graph 基线、当前系统（32 题×4 能力）：用数据发现'盲目加知识图谱 + 错误路由'使召回从 0.59 跌到 0.02（反不如 vanilla RAG）；诊断根因后，用目标 Cypher 修复 graph_rag（关系召回 0→0.95）+ 数据驱动路由，overall Recall@5 0.02→0.84、MRR 0.05→0.86。"* —— 完整的"发现问题→定位→修复→量化"闭环，且自带"反直觉发现"（加图谱不一定更好）这个面试记忆点。

## P3 前端可视化（检索工坊，impeccable 设计）

### P3-1 工坊 SPA：对话 / 检索溯源 / 知识图谱 / 监控 ⭐⭐
- **设计（impeccable）**：选"工坊"方向（product register，给开发者/面试官），vanilla HTML/CSS/JS（无构建、原生）。PRODUCT.md 固化气质=transparent/precise/credible，参照 Linear/Vercel/Grafana。
- **配色（OKLCH）**：纯白底 + 单赤陶点缀 `oklch(0.59 0.188 36)`（呼应"烹饪"暖意，但暖意只在点缀色、不在底——刻意避开 2026 AI 默认的奶油底）；图谱食材用蓝绿 `oklch(0.62 0.085 200)` 与赤陶色盲可分；节点同时用颜色+形状+标签三重编码。
- **三栏布局**：
  - 左·对话：POST `/api/ask`，流式骨架态、路由/缓存/延迟 chip、拒答专用态。
  - 中·检索溯源 X 光：路由策略 + 缓存命中层 + **每步耗时条**（gate/retrieve/generate，把 P5"瓶颈在生成"直接可视化）+ 命中菜谱表（每路来源 vector/bm25/dual/graph + RRF 分数）。
  - 右·知识图谱：POST `/api/graph` 返回命中菜谱的 1 跳子图（Recipe–REQUIRES–Ingredient + Category），SVG 确定性二分图（菜谱左/食材右，共享食材自然成 hub）。
  - 顶·监控条：轮询 `/api/stats`（QPS/p99/缓存命中率/路由分布），小功能 chip 而非 SaaS 大数字俗套。
- **后端配套**：`/api/stats`（轻量 RequestStats：滚动窗口 QPS/p50/p95/p99 + 缓存分层 + 路由分布 + 拒答）；`/api/graph`（`graph_rag_retrieval.provenance_graph` 1 跳 Cypher）；`ask_async` 增加 `stages`（gate/retrieve/generate ms）返回；FastAPI 挂 `/static` + `GET /`。
- **impeccable 合规**：无侧边色条、无渐变文字、无玻璃拟态、无 SaaS 大数字模板、无统一卡片网格、无全大写小标 eyebrow；正文对比度 ≥4.5:1；`prefers-reduced-motion` 降级；色盲三重编码。
- **验证**：`GET /`→HTML、`/static/*`→200、`/api/stats` JSON、`/api/graph` 返回节点/边（REQUIRES+CATEGORY）、`/api/ask` 返回 stages+sources（关系题 graph_rag+graph_relation 源、lookup 题 bm25/dual/vector 多路）；app.js `node --check` 通过、9 个 DOM id 全匹配。
- **涉及文件**：`static/{index.html,styles.css,app.js}`、`core/stats.py`、`api/server.py`、`rag_modules/graph_rag_retrieval.py`、`main.py`。
- **查看**：服务起后浏览器开 `http://localhost:8000`。
- **后续**：用户实际查看后可跑 `/impeccable critique` 打分 + `polish` 收尾。
